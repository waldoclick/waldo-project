  <name>Task 1: Fire purchase event in pagar/gracias.vue before clearAll</name>
  <files>apps/website/app/pages/pagar/gracias.vue</files>
  <action>
    Edit `apps/website/app/pages/pagar/gracias.vue` to fire the purchase event using order data.

    **Key constraint:** `adStore.clearAll()` in `onMounted` wipes analytics state. The purchase event MUST fire before clearAll, and MUST use `orderData.value` (the fetched order), NOT store analytics state.

    **The `purchaseFired` ref already exists** (line 40). Use it as a one-shot guard.

    **Problem:** `orderData` is a computed that depends on `data` from `useAsyncData`. On `onMounted`, the async data may already be loaded (SSR hydration) or still pending. Use a `watchEffect` or `watch` that fires when `orderData.value` becomes non-null AND `purchaseFired.value` is false.

    **Implementation:** Replace the existing `onMounted` with:

    ```typescript
    // Fire purchase event once when order data is available, BEFORE clearing store
    watch(
      orderData,
      (order) => {
        if (order && !purchaseFired.value) {
          purchaseFired.value = true;
          adAnalytics.purchase(order);
        }
      },
      { immediate: true },
    );

    onMounted(() => {
      adStore.clearAll();
    });
    ```

    Using `watch(orderData, ..., { immediate: true })` means:
    - On SSR hydration, if `orderData` is already populated, fires immediately
    - If data arrives after mount, fires when the computed resolves
    - The `purchaseFired` guard prevents double-firing on reactive re-evaluation

    The `watch` must be imported from vue (it's already in the import line: `import { computed, watchEffect, ref, onMounted } from "vue"` — add `watch` to this import).

    The `OrderData` interface in the file is compatible with `PurchaseOrderData` (it has `documentId`, `amount`, `totalAmount`, `currency`, `payment_response.buy_order`). TypeScript cast: `adAnalytics.purchase(order as PurchaseOrderData)` if needed, or just pass `order` directly since the fields overlap.

    Import `PurchaseOrderData` from the composable if a cast is needed:
    ```typescript
    import { useAdAnalytics, type PurchaseOrderData } from "~/composables/useAdAnalytics";
    ```

    **Pack-only flow detection:** `adStore.ad.ad_id === null`. When a user comes from /packs, `ad_id` is null because they haven't created an ad — they're buying a pack standalone.

    **Why here and not in CheckoutDefault.vue:** The component fires `addPaymentInfo` but never `begin_checkout` for pack-only flow. `anunciar/resumen.vue` fires `begin_checkout` for the ad-creation flow in its own `onMounted`. To mirror this pattern for pack flow, fire it on the page that is the entry point (`/pagar`).

    **Implementation:** Add the following to `pagar/index.vue`:

    ```typescript
    import { onMounted } from "vue";
    import { useAdStore } from "@/stores/ad.store";
    import { useAdAnalytics } from "@/composables/useAdAnalytics";

    const adStore = useAdStore();
    const adAnalytics = useAdAnalytics();

    onMounted(() => {
      // begin_checkout for pack-only flow (ad_id is null when coming from /packs)
      // For ad-creation flow, begin_checkout already fires in anunciar/resumen.vue
      if (adStore.ad.ad_id === null) {
        adAnalytics.beginCheckout();
      }
    });
    ```

This is acceptable — if items are empty, `beginCheckout` skips the push (existing behavior: `if (items.length > 0) pushEvent(...)`). The event silently no-ops if no items are in analytics, which is correct behavior rather than sending an empty/undefined event. Document this with a comment.
