# Phase 24 Plan 01: Implement featuredCron + Commit cron-runner

## Objective

Implement `featured.cron.ts` with `FeaturedCronService.restoreFreeFeaturedReservations()` that
guarantees every user always has 3 free available `ad-featured-reservation` slots. Register
`featuredCron` in `cron-tasks.ts` at 2:30 AM America/Santiago. Commit the existing untracked
`cron-runner` API files (controller + routes).

## Context

- **Schema**: `ad-featured-reservation` has `price` (biginteger, required), `total_days` (integer, optional), `user` (manyToOne → users-permissions.user), `ad` (oneToOne → api::ad.ad), `description` (text). No `draftAndPublish` — no `publishedAt` required.
- **"Free available" definition**: `price = 0` AND (`ad = null` OR `ad.active = false`)
- **Existing pattern**: `user.cron.ts` → `UserCronService` class; same structure to follow
- **Existing utility**: `PaymentUtils.general.ensureFreeFeaturedReservations()` in `general.utils.ts` creates reservations without the availability check — the cron must do its own count-then-create logic
- **cron-runner files** already authored and working (untracked): `apps/strapi/src/api/cron-runner/controllers/cron-runner.ts` and `apps/strapi/src/api/cron-runner/routes/cron-runner.ts`. The controller already has `"featured-cron": "featuredCron"` in its name map.
- **Schedule gap**: existing crons run at 1:00 AM (adCron), 2:00 AM (userCron), 3:00 AM (backupCron), 4:00 AM Sunday (cleanupCron). `featuredCron` at 2:30 AM fits cleanly.
- **No `total_days`**: Free featured reservations have no expiry concept — `total_days` must NOT be set (leave undefined/omitted). This differs from `ad-reservation` which always sets `total_days: 15`.
- **Logger**: use `import logger from "../utils/logtail"` — same as `user.cron.ts`
- **Strapi entityService**: use `strapi.entityService.findMany` and `strapi.entityService.create` — same as `user.cron.ts`

## Tasks

### Task 1: Create `featured.cron.ts`
**type**: auto

Implement `apps/strapi/src/cron/featured.cron.ts` as a class-based service matching the `user.cron.ts` pattern.

**Implementation**:

```typescript
import logger from "../utils/logtail";

export interface ICronjobResult {
  success: boolean;
  results?: string;
  error?: string;
}

export default class FeaturedCronService {
  async restoreFreeFeaturedReservations(): Promise<ICronjobResult> {
    // 1. Guard: check strapi is available
    // 2. Log start
    // 3. Fetch all users (plugin::users-permissions.user, pagination pageSize: -1, only id field needed)
    // 4. For each user:
    //    a. Count their free available featured reservations:
    //       - findMany api::ad-featured-reservation.ad-featured-reservation
    //       - filters: user.id = userId, price = 0, $or: [{ ad: null }, { ad: { active: false } }]
    //       - populate: { ad: true }
    //    b. neededSlots = Math.max(0, 3 - freeAvailableCount)
    //    c. For each missing slot: create with price=0, user=userId, description, publishedAt
    //       (NO total_days — featured reservations have no expiry)
    // 5. Log summary (total users processed, total slots created)
    // 6. Return { success: true, results: "..." }
  }
}
```

**Key implementation notes**:
- `price` is a biginteger in the schema. When filtering, use `{ price: { $eq: "0" } }` or `{ price: 0 }` — test which Strapi v5 entityService accepts. Looking at `user.cron.ts`, it uses `price: 0` (number) in filters successfully for `ad-reservation`, so use the same.
- For the `$or` filter with nested `ad.active = false`, use:
  ```typescript
  $or: [
    { ad: null },
    { ad: { active: { $eq: false } } }
  ]
  ```
- Populate `ad: true` to get the `active` field for filtering in JS (post-fetch check) if the nested filter proves unreliable — but prefer the DB filter approach first.
- `description`: `Free featured reservation restored ${new Date().toISOString()}`
- `publishedAt: new Date()` — required for Strapi to treat it as published
- Track `totalCreated` across all users for the summary log
- Per-user errors should be caught and logged (same try/catch-per-user pattern as `user.cron.ts`) — don't fail the whole cron for one user

**English JSDoc and inline comments required**:
- Class-level JSDoc: purpose, guarantee, what "free available" means
- Method-level JSDoc: algorithm overview (scan → count → create)
- Inline comments on: guard check, user query, free-available filter, neededSlots calc, create loop

**Done criteria**: File exists at `apps/strapi/src/cron/featured.cron.ts`, class is exported as default, method signature matches, TypeScript compiles without errors.

### Task 2: Register `featuredCron` in `cron-tasks.ts`
**type**: auto

Add `featuredCron` entry to `apps/strapi/config/cron-tasks.ts`:

1. Add import: `import FeaturedCronService from "../src/cron/featured.cron";`
2. Add job entry following the established JSDoc pattern from v1.7:

```typescript
/**
 * Ensures every user has exactly 3 free available ad-featured-reservation records
 * (price = 0, ad = null or ad.active = false). Creates missing slots for any user
 * who has fewer than 3. Runs daily at 2:30 AM Santiago time (America/Santiago).
 * Calls FeaturedCronService.restoreFreeFeaturedReservations() — see featured.cron.ts for full logic.
 */
featuredCron: {
  task: async ({ strapi }) => {
    strapi.log.info("=== STARTING FREE FEATURED RESERVATION RESTORE ===");
    const featuredCronService = new FeaturedCronService();
    await featuredCronService.restoreFreeFeaturedReservations();
    strapi.log.info("=== FREE FEATURED RESERVATION RESTORE COMPLETE ===");
  },
  options: {
    rule: "30 2 * * *", // Every day at 2:30 AM (America/Santiago)
    // rule: "* * * * *", // Test
    tz: "America/Santiago",
  },
},
```

**Done criteria**: `cron-tasks.ts` imports `FeaturedCronService`, has `featuredCron` entry with correct schedule (`"30 2 * * *"`), TypeScript compiles.
