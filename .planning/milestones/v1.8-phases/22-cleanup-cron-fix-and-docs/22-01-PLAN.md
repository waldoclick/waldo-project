Fix `getStrapiImages()` in `CleanupService` so it correctly retrieves files from the `ads` folder.

**Root cause:** `entityService.findMany('plugin::upload.file', { filters: { folder: { name: 'ads' } } })` silently returns an empty array in Strapi v5 because the `folder` relation filter path is not supported on plugin content types via the entityService filter API.

**Fix approach — use `folderPath` string filter:**
`plugin::upload.file` has a `folderPath` string attribute that contains the folder's path string (e.g. `"/1"` or `"/ads"`). Filter on this directly instead of traversing the relation:

```ts
const images = await strapi.entityService.findMany(
  "plugin::upload.file",
  {
    filters: {
      folderPath: {
        $startsWith: "/", // all files in any folder — then refine
      },
    },
    pagination: { pageSize: -1 },
  }
);
```

However, `folderPath` stores the numeric path (e.g. `/1`), not the folder name. The correct approach is to **first resolve the folder's path, then filter files by it**:

1. Query the `ads` folder to get its `path` string:
```ts
const adsFolder = await strapi.db
  .query("plugin::upload.folder")
  .findOne({ where: { name: "ads" } });
```

2. If `adsFolder` is null, log a warning and return `[]` (folder does not exist yet — safe to skip).

3. Filter files by `folderPath` equal to `adsFolder.path`:
```ts
const images = await strapi.entityService.findMany(
  "plugin::upload.file",
  {
    filters: { folderPath: adsFolder.path },
    pagination: { pageSize: -1 },
  }
);
```

This two-step approach is the correct Strapi v5 pattern for filtering upload files by folder name.
