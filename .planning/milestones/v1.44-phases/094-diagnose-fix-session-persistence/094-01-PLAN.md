<objective>
Fix the session persistence bug where `guard.global.ts` redirects authenticated dashboard users
to `/auth/login` after a page refresh following the 2-step verify-code flow.

Purpose: The root cause is the `@nuxtjs/strapi` plugin (`strapi.js`) running `fetchUser()` on every
page load — including SSR. If `/users/me` fails on SSR (timeout, heavy populate, Strapi not ready),
the catch block calls `setToken(null)`, actively destroying the JWT cookie before the guard runs.
This plan fixes that by preventing the destructive SSR `fetchUser` and making the guard cookie-aware.

Output:
- `apps/dashboard/app/plugins/strapi-fetch-user.client.ts` — replaces the SSR fetch with client-only
- Updated `guard.global.ts` — adds token presence check so a cookie-holding user is never pre-emptively redirected on SSR
</objective>

**Root cause (SESS-01 documentation):** The `@nuxtjs/strapi` module registers `plugins/strapi.js` which
runs on BOTH server and client. On SSR (page refresh), it calls `fetchUser()`. The `fetchUser()` implementation
wraps `/users/me` in a try/catch where ANY error calls `setToken(null)` — actively destroying the JWT cookie.
The dashboard's heavy `auth.populate` (`ad_reservations.ad`, `ad_featured_reservations.ad`) makes `/users/me`
slow and error-prone on SSR. The destroyed cookie means `guard.global.ts` sees `user.value = null` and
redirects to `/auth/login`, even though the user legitimately authenticated.

**Step 1 — Create `apps/dashboard/app/plugins/strapi-fetch-user.client.ts`:**

This plugin runs ONLY on the client (`.client.ts` suffix), replacing the SSR-triggered fetch.
Since Nuxt loads plugins alphabetically and the module's `strapi.js` is registered by the module (before app plugins),
we need to use the `enforce: 'post'` trick or rely on the fact that app plugins run after module plugins.
Actually the cleanest fix is: the module's plugin runs server+client. Our client plugin runs after it.
BUT the problem is on SSR — the module plugin runs server-side and nukes the token.

The correct fix: **Disable or neutralize SSR fetchUser by overriding the strapi plugin behavior at the server level.**
Use a Nuxt server plugin (`.server.ts`) that sets `user.value` to a non-null sentinel from the cookie alone,
preventing the module plugin from calling `fetchUser()` (it only calls `fetchUser` when `user.value` is null).

Wait — re-reading the module plugin:
```js
export default defineNuxtPlugin(async () => {
  const user = useStrapiUser();
  if (!user.value) {      // ← only fetches if user is null
    const { fetchUser } = useStrapiAuth();
    await fetchUser();
  }
});
```

On SSR after a hard refresh: `user.value` IS null (it's a fresh SSR request), `token.value` IS present
(from the `waldo_jwt` cookie in the request headers). So `fetchUser()` IS called, hits `/users/me`, and if
that call fails for any reason — `setToken(null)` fires.

**The fix: Create a `plugins/strapi-fetch-user.server.ts`** that runs BEFORE the module plugin on server,
reads the token from the cookie, and if a token is present, marks a server state so that the guard can skip
the redirect. Actually the ordering is: Nuxt registers module plugins first, then app plugins. So a server
plugin in `app/plugins/` runs AFTER the module's `strapi.js`. This means the nuke happens before our plugin.

**True fix — two-pronged approach:**

**Prong A — Guard becomes token-aware (the key fix):**
Update `guard.global.ts` to check `useStrapiToken().value` as a fallback when `user.value` is null.
If a token exists in the cookie, DO NOT redirect to login on SSR — let the client hydrate and the
client-side `fetchUser()` will populate `user.value` properly. This is safe because:
- On SSR, if the token is present but `/users/me` failed, we allow the page to render (no redirect)
- On client hydration, `fetchUser()` runs again and either populates `user.value` or destroys the token
- The existing `if (!roleName) return` guard already handles the SSR partial-user case

**Prong B — Safer fetchUser wrapper (prevents future token destruction):**
Create `apps/dashboard/app/plugins/strapi-fetch-user.client.ts` that replaces the client-side fetch
with a version that does NOT call `setToken(null)` on error — it silently leaves the token intact and
lets the guard/user decide what to do. This is done via a Nuxt plugin that overwrites `provide` behavior.

Actually, the simplest and most robust fix is **Prong A alone**: guard checks token before redirecting.

**Implementation:**

**File 1: `apps/dashboard/app/middleware/guard.global.ts`** — add token fallback:

```ts
import type { User } from "@/types/user";

export default defineNuxtRouteMiddleware((to, _from) => {
  const publicRoutes = [
    "/auth/login",
    "/auth/forgot-password",
    "/auth/reset-password",
    "/auth/verify-code",
    "/dev",
  ];

  const isPublicRoute = publicRoutes.some((route) => to.path.startsWith(route));
  if (isPublicRoute) return;

  const user = useStrapiUser() as Ref<User | null>;

  // On SSR: useStrapiAuth fetchUser() may fail and call setToken(null), leaving user null
  // even when a valid waldo_jwt cookie exists. Check the token directly as a safety net:
  // if a token is present, allow the request through and let client-side hydration restore the user.
  if (!user.value) {
    const token = useStrapiToken();
    if (token.value) {
      // Token is present — fetchUser likely failed on SSR (heavy populate, timeout, Strapi not ready).
      // Allow SSR render; client-side plugin will call fetchUser() and resolve user state correctly.
      return;
    }
    useCookie("redirect", { path: "/" }).value = to.fullPath;
    return navigateTo("/auth/login");
  }

  const userRole = user.value?.role;
  const roleName = userRole?.name?.toLowerCase() || null;

  // Role is only populated after /users/me is called (client-side).
  // On SSR the JWT is present but role is undefined — skip the role check to
  // avoid a redirect loop that crashes the server with OOM.
  if (!roleName) return;

  if (roleName !== "manager") {
    // Non-manager authenticated user — not allowed in the dashboard.
    // Redirect to login without calling useLogout() (unsafe in middleware context).
    // The website session cookie is intentionally left untouched.
    return navigateTo("/auth/login");
  }
});
```

**File 2: `apps/dashboard/app/plugins/strapi-fetch-user.client.ts`** — client-side fetchUser that
does NOT destroy the token on error. This replaces the module's client-side fetch behavior by running
after the module plugin and re-fetching if user is still null but token is present:

```ts
// Client-only plugin: ensures fetchUser() does NOT destroy the token on /users/me error.
// The @nuxtjs/strapi module plugin runs before this one. If SSR already populated user.value,
// this is a no-op. If SSR failed and nuked the token, we cannot recover here — but the guard's
// token check (Prong A) already prevented the redirect, so the page rendered. Here we try to
// restore user.value on client hydration using the cookie token if it's still present.
import { defineNuxtPlugin } from "#imports";

export default defineNuxtPlugin(async () => {
  const user = useStrapiUser();
  const token = useStrapiToken();

  // Only attempt if user not already populated but token exists
  // (the module plugin may have already populated user.value — this is then a no-op)
  if (!user.value && token.value) {
    try {
      const { fetchUser } = useStrapiAuth();
      await fetchUser();
    } catch {
      // Intentionally swallow: fetchUser's internal catch already handles token state.
      // We do not call setToken(null) here — the module's fetchUser already does that
      // if /users/me truly fails. If it succeeded, user.value is now populated.
    }
  }
});
```

**Important:** The `.client.ts` suffix means this plugin runs client-side only. Combined with the guard
fix, this ensures: SSR never redirects a cookie-holding user; client hydration populates user correctly.

**Regression check:**
- Non-manager users: `roleName !== "manager"` check still runs after `fetchUser()` populates role client-side
- No-token users (truly unauthenticated): `token.value` is null → redirect fires as before
- Password reset / forgot-password: public routes bypass guard entirely — unaffected
- OAuth callback: unaffected (separate flow)
- `useLogout()`: calls `strapiLogout()` + `setToken(null)` → token gone → next guard check redirects correctly
