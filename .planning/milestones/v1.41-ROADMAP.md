# Milestone v1.41: Ad Preview Error Handling

**Status:** ✅ SHIPPED 2026-03-18
**Phases:** 093
**Total Plans:** 2

## Overview

Eliminates the 500 error that occurred when previewing an ad from the dashboard. Root cause was a `watchEffect` + `showError()` pattern in `[slug].vue` that fired during SSR and caused a fatal render crash instead of delegating cleanly to `error.vue`. Also hardened the Strapi `findBySlug` controller against unexpected DB errors. The fix is purely subtractive in the website and additive (try/catch + tests) in Strapi.

## Phases

### Phase 093: Ad Preview Error Handling

**Goal**: The website never returns 500 on `/anuncios/[slug]` — errors propagate cleanly as 404/5xx Nuxt error pages
**Depends on**: Nothing (standalone bug fix)
**Requirements**: PREV-01, PREV-02, PREV-03, PREV-04, STRP-01
**Plans**: 2 plans

Plans:

- [x] 093-01: Strapi findBySlug try/catch + Jest test (TDD RED→GREEN)
- [x] 093-02: `[slug].vue` createError refactor + TypeScript check + smoke test

**Details:**

**093-01 (Strapi):** Added `try/catch` around `strapi.service("api::ad.ad").findBySlug(slug, userId)` in `apps/strapi/src/api/ad/controllers/ad.ts`. The catch block calls `strapi.log.error("findBySlug error for slug %s: %o", slug, error)` and returns `ctx.internalServerError("Internal server error")` — zero stack trace exposed. The JWT decode block (already had its own inner try/catch) was intentionally kept outside the new outer try. 4 Jest tests added: null→notFound, throw→internalServerError, happy path manager, happy path public.

**093-02 (Website):** 6 surgical changes in `apps/website/app/pages/anuncios/[slug].vue`:
1. Removed `pending,` from `useAsyncData` destructuring (unused after refactor)
2. Removed `error: adError,` from destructuring (unused after refactor)
3. Replaced `return null` (null result path) with `throw createError({ statusCode: 404, fatal: true, message: 'Página no encontrada', description: 'El aviso que buscas no existe o no está disponible.' })`
4. Added `default: () => null` to `useAsyncData` options
5. Deleted `getErrorMessage()` function entirely
6. Deleted `watchEffect(() => { if (!pending.value && !adData.value) { showError(getErrorMessage()) } })` entirely

Positive deviation: catch block re-throws Nuxt errors (prevents the 404 path from being incorrectly converted to a 500). TypeCheck clean. Smoke test approved by user.

## Milestone Summary

**Key Decisions:**

- `createError({ fatal: true })` inside `useAsyncData` is the Nuxt 4 canonical pattern for SSR-safe error pages — `showError()` in `watchEffect` is an anti-pattern that fires during SSR setup and causes a 500 crash
- Catch block re-throws Nuxt errors before the outer `createError({ statusCode: 500 })` — prevents the 404 path from being swallowed by the generic error handler
- TDD order: Jest test scaffold (RED) before implementation (GREEN) — explicitly confirmed the behavior was broken before fixing it

**Issues Resolved:**

- 500 error on `/anuncios/[slug]` in all environments (local dev and production) when any error occurred — including 404 from non-existent or non-accessible ads
- Session loss on website after manager previews ad from dashboard (the 500 was clearing the cookie)
- Missing `default: () => null` in `useAsyncData` (AGENTS.md compliance)

**Issues Deferred:**

- POLL-01: Post-logout website Pinia stores reset when logout originates from dashboard (minor stale-data UX; no security risk)
- Staging cross-domain verification: `COOKIE_DOMAIN=.waldoclick.dev` smoke test (code is correct, env var must be set)

**Technical Debt Incurred:**

- None — purely subtractive in the website, TDD-backed in Strapi
