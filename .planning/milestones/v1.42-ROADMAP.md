# Milestone v1.42: Dashboard Session Persistence

**Shipped:** 2026-03-18
**Phase:** 094

## Goal

Dashboard users who log in through the 2-step verify-code flow remain authenticated after a page refresh — the guard never redirects an authenticated user to login.

## Requirements

| Requirement | Description | Status |
|-------------|-------------|--------|
| SESS-01 | Root cause documented | ✅ Complete |
| SESS-02 | `waldo_jwt` cookie persists between reloads | ✅ Complete |
| SESS-03 | Guard does not redirect authenticated user on refresh | ✅ Complete |
| SESS-04 | Full login → verify-code → refresh flow works | ✅ Complete |

## Root Cause

`@nuxtjs/strapi`'s `strapi.js` module plugin runs on every SSR page load. When `user.value` is null, it calls `fetchUser()` which calls `/users/me`. The catch block silently calls `setToken(null)` on any error — destroying the JWT cookie before `guard.global.ts` runs.

The `auth.populate` array included `"ad_reservations.ad"` and `"ad_featured_reservations.ad"` — heavy relational joins not present in the `User` TypeScript interface and not consumed by any dashboard component. These caused the SSR `/users/me` query to fail, triggering the token destruction and the redirect-to-login bug.

## Fix

**`apps/dashboard/nuxt.config.ts`** — removed unused populate fields from `strapi.auth.populate`:
- Removed: `"ad_reservations.ad"`, `"ad_featured_reservations.ad"`
- Remaining: `"role"`, `"commune"`, `"region"`, `"business_region"`, `"business_commune"`

**`apps/dashboard/.env`** — corrected `SESSION_MAX_AGE` from `604800000` (ms) to `604800` (seconds).
