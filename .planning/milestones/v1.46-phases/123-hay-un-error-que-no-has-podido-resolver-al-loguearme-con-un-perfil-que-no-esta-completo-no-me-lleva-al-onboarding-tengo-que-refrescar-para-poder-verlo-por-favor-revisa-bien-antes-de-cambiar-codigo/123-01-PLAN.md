<objective>
Fix the bug where logging in with an incomplete profile (email+OTP, Google, or Facebook) does not auto-navigate to /onboarding â€” the user has to refresh the page manually for the redirect to fire.

Purpose: The global `onboarding-guard.global.ts` middleware already correctly redirects incomplete-profile users to `/onboarding`. The bug is that `FormVerifyCode.vue` and `login/google.vue` use `router.push()` (raw Vue Router) instead of `navigateTo()` (Nuxt-aware). In Nuxt 4 `router.push` can silently abort or race with the middleware pipeline, especially after async `fetchUser()` state updates. Switching to `navigateTo()` â€” the same primitive used in `useLogout.ts` and the guard itself â€” fixes the timing race so the guard fires correctly on the post-login navigation.

Output: Two components migrated from `router.push`â†’`navigateTo`, one stale test file corrected. No guard logic changes, no store changes, no new files.
</objective>

```typescript
const AUTH_EXEMPT_PATHS = new Set(["/login", "/registro", "/logout"]);

export default defineNuxtRouteMiddleware(async (to) => {
  if (AUTH_EXEMPT_PATHS.has(to.path)) return;

  const user = useStrapiUser();
  if (!user.value) {
    const token = useStrapiToken();
    if (!token.value) return;
    const { fetchUser } = useStrapiAuth();
    await fetchUser();
  }
  if (!user.value) return;

  const meStore = useMeStore();
  const profileComplete = await meStore.isProfileComplete();

  if (!profileComplete) {
    if (to.path.startsWith("/onboarding")) return;
    return navigateTo("/onboarding");
  }

  if (to.path === "/onboarding") {
    return navigateTo("/");
  }
});
```

Key facts from current guard:

2. GUARD-02 uses STRICT equality (`to.path === "/onboarding"`), NOT `startsWith`. So a complete user at `/onboarding/thankyou` is NOT redirected.
3. When `user.value` is null the guard reads `useStrapiToken()` and calls `useStrapiAuth().fetchUser()`. Tests that return `{ value: null }` for user must mock those two globals or the code path will throw.
