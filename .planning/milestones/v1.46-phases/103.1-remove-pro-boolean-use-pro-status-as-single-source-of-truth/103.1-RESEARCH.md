# Phase 103.1: Remove pro boolean — use pro_status as single source of truth - Research

**Researched:** 2026-03-21
**Domain:** Strapi user schema refactor + frontend boolean-to-enum migration
**Confidence:** HIGH

## Summary

The `pro` boolean field on the User schema is a legacy flag that was added before `pro_status` (enum: `active`/`inactive`/`cancelled`) and `pro_expires_at` were introduced for the Oneclick subscription system. Currently both fields coexist on the user record and must be kept in sync manually. This dual-source-of-truth is a bug risk: code that writes one but not the other will produce inconsistent state.

The goal of this phase is to **remove all reads and writes of `user.pro`** from application code and replace them with `user.pro_status === "active"` as the single source of truth. The `pro` boolean field itself will remain in the Strapi schema temporarily as a compatibility shim (Strapi v5 cannot remove columns without a migration), but no application code should read or write it after this phase. If desired, the field can be fully dropped from the schema at the end of this phase with a DB migration step.

**Primary recommendation:** Replace all `user.pro` boolean reads with `user.pro_status === "active"` computed checks. Remove all `pro: true/false` writes from server-side Strapi updates. Update type definitions to remove or deprecate the `pro` field. Update `computeSortPriority` to use `pro_status` instead of `pro`.

## Architecture Patterns

### Pattern: Derive PRO status from `pro_status` enum, not from `pro` boolean

**What:** `pro_status: "active" | "inactive" | "cancelled"` is the authoritative field. A user is PRO if and only if `pro_status === "active"`.

**Helper pattern (website):** A computed or inline check:
```typescript
// Instead of: user.pro
// Use:
const isPro = computed(() => user.value?.pro_status === "active");
```

**Helper pattern (Strapi):** In `computeSortPriority`, change the type annotation and check:
```typescript
// Before:
export function computeSortPriority(ad: {
  user?: { pro?: boolean } | null;
}): number {
  const isPro = !!(ad.user && (ad.user as { pro?: boolean }).pro);
  return isPro ? 0 : 1;
}

// After:
export function computeSortPriority(ad: {
  user?: { pro_status?: string } | null;
}): number {
  const isPro = !!(ad.user && (ad.user as { pro_status?: string }).pro_status === "active");
  return isPro ? 0 : 1;
}
```

### Pattern: Sanitize-ad must expose `pro_status`, not `pro`

The `sanitizeAdForPublic` function builds the safe user object served via the public API. It currently exposes `pro: user.pro`. After this change, it should expose `pro_status: user.pro_status` instead. The frontend `CardAnnouncement.vue` reads `getUser.pro` — this must be updated in the same commit to read `getUser.pro_status === "active"`.

### Pattern: Server-side writes omit `pro` entirely

No server-side code should write `pro: true` or `pro: false` after this phase. The Strapi entity service calls in `proResponse` and `chargeExpiredSubscriptions` must drop the `pro` field from their update payloads.

### Pattern: DB schema field remains but is ignored

The `pro` field in `schema.json` stays. Removing it requires a Strapi migration (ALTER TABLE DROP COLUMN), which is out of scope for this phase unless explicitly requested. The field simply becomes a stale column that no longer gets written. Existing `true` values in the DB will remain but be ignored.

**Alternative (if full removal desired):** Set all rows to `false`/null in a one-time migration SQL, then remove from `schema.json` and restart Strapi. This is clean but requires downtime care. Scope this as a separate optional step.

|---------|-------------|-------------|

| Keeping `pro` in sync | Write `pro` alongside `pro_status` | Remove `pro` writes entirely — `pro_status` is the source |
| Type-safe PRO check in TS | Type guard function | TypeScript union type on `pro_status` field is sufficient |

## Common Pitfalls

### Pitfall 1: Updating server code but not sanitize-ad
**What goes wrong:** `computeSortPriority` correctly uses `pro_status`, but `sanitizeAdForPublic` still returns `pro: user.pro` (false always, since it's no longer written). Frontend `CardAnnouncement.vue` reads `getUser.pro` — always sees `false` — seller link disappears for all PRO users.
**How to avoid:** Always update `sanitizeAdForPublic` + `CardAnnouncement.vue` in the same task/commit.

### Pitfall 2: Type cast `{ pro?: boolean }` left in place after logic change
**What goes wrong:** `computeSortPriority` logic is updated to use `pro_status`, but the TypeScript cast `{ pro?: boolean }` in call sites (`lifecycles.ts`, `payment.ts`, `subscription-charge.cron.ts`) still compiles because `pro` is still in the schema. TypeScript will not catch the inconsistency.
**How to avoid:** Update all cast annotations to `{ pro_status?: string }` in the same pass as the logic change.

### Pitfall 3: Dashboard admin panel loses PRO visibility
**What goes wrong:** `apps/dashboard/app/pages/users/[id].vue` shows `formatBoolean(item.pro)`. After `pro` is no longer written, all users show "No" for PRO — admin cannot see who is PRO.
**How to avoid:** Update dashboard display to show `item.pro_status` (string value, not formatted boolean).

### Pitfall 4: Website PRO gates use `!user.pro` — all PRO users get 403
**What goes wrong:** Pages `avatar.vue`, `cover.vue`, `username.vue` gate with `if (!user.value?.pro)`. Once `pro` is removed from the type and API response, `user.value?.pro` is `undefined` — falsy — all users get 403 even if PRO.
**How to avoid:** Update gates before deploying. The gate change must be in the same deployment as the sanitize-ad change.

### Pitfall 5: `protect-user-fields.ts` still strips `"pro"` from requests
**What goes wrong:** This is low-risk (stripping a field that no longer matters is harmless), but it's confusing dead code that references a deprecated field.
**How to avoid:** Remove `"pro"` from the `PROTECTED_USER_FIELDS` array and update the test.

### Pitfall 6: `recalculateSortPriorities()` bootstrap function also uses the cast
**What goes wrong:** `ad.ts` also has `recalculateSortPriorities()` which calls `computeSortPriority` with the same cast pattern. This must be updated alongside `computeSortPriority`.
**How to avoid:** Grep confirms `computeSortPriority` is defined once and called in multiple places — updating the function signature handles all callers automatically.

## Open Questions

1. **Should the `pro` field be dropped from `schema.json` in this phase?**
   - What we know: Strapi v5 supports removing fields from the schema JSON. On restart, Strapi runs a DB sync that may or may not drop the column (depends on `syncDatabase` config).
   - What's unclear: The project's Strapi DB sync configuration. If `syncDatabase: false` (production mode), removing from schema.json does NOT drop the column. The column stays as dead weight.
   - Recommendation: Keep `pro` in `schema.json` for this phase (zero risk). Add a `// DEPRECATED: use pro_status` comment. Plan a separate optional cleanup migration if desired.

2. **Does `useStrapiUser()` return `pro_status` from the `/users/me` endpoint?**
   - What we know: `pro_status` is a non-private field on the user schema — Strapi's `/users/me` endpoint returns all non-private fields by default.
   - What's unclear: Whether the existing `users-permissions` middleware that adds `role` and `commune` to `/users/me` also needs updating. Check `apps/strapi/src/extensions/users-permissions/`.
   - Recommendation: Verify by checking what the `/users/me` middleware returns. Almost certainly `pro_status` is already in the response since it was added to the schema in Phase 102.
