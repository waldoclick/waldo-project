# Quick Task 260504-ue0: Email Confirmation Auto-Resend + SSR Verification

<objective>
Two coupled changes:

1. Strapi `overrideEmailConfirmation`: when the token belongs to an already-confirmed user, auto-generate a new confirmationToken, persist it, send a fresh MJML confirmation email, and return `{ resent: true }` instead of `badRequest("Token already used")`.

2. `apps/website/app/pages/registro/activar.vue`: refactor to be SSR-compatible by using `useAsyncData` (per CLAUDE.md rule: "useAsyncData() is the sole correct data-loading trigger in Nuxt pages"), with three-outcome handling for `{ ok: true }`, `{ resent: true }`, and error.

Purpose: stop dropping users onto a dead-end "Token already used" error. If a user clicks an old link, send them a brand new one transparently. SSR-compatible page eliminates hydration flash.

Output: One Strapi controller modification and one Nuxt page rewrite — both behavior-preserving for the success path, additive for the resend branch.
</objective>

Concrete edits:

1. Expand the `select` array in the user lookup so we have the data needed to send an email and surface the recipient:
   ```ts
   const user = await strapi.db.query("plugin::users-permissions.user").findOne({
     where: { confirmationToken },
     select: ["id", "email", "username", "firstname", "confirmed", "blocked"],
   });
   ```

2. Keep the early-return for missing/blocked user UNCHANGED:
   ```ts
   if (!user || user.blocked) {
     return ctx.badRequest("Invalid confirmation token");
   }
   ```
   (No email sent — we don't have a target address to resend to in this case. The task spec is explicit: "can't resend, no email".)

3. Replace the existing `if (user.confirmed)` block (which currently nulls the token and returns `badRequest("Token already used")`) with the auto-resend behavior:
   ```ts
   if (user.confirmed) {
     // Generate a fresh token (same pattern as overrideForgotPassword line ~486)
     const newConfirmationToken = crypto.randomBytes(64).toString("hex");

     await strapi.db.query("plugin::users-permissions.user").update({
       where: { id: user.id },
       data: { confirmationToken: newConfirmationToken },
     });

     const confirmationUrl = `${
       process.env.FRONTEND_URL || "http://localhost:3000"
     }/registro/activar?confirmation=${newConfirmationToken}`;
     const name = user.firstname || user.username || user.email;

     try {
       await sendMjmlEmail(
         strapi,
         "email-confirmation",
         user.email,
         "Confirma tu correo electrónico",
         { name, confirmationUrl }
       );
     } catch (err) {
       // Non-fatal — token is already saved; user can retry sendEmailConfirmation
       strapi.log.error(
         `[overrideEmailConfirmation] Failed to resend confirmation email to ${user.email}: ${err?.message ?? err}`
       );
     }

     return ctx.send({ resent: true });
   }
   ```

4. Leave the unconfirmed branch (the current final `update` + `ctx.send({ ok: true })`) UNCHANGED.

Why this approach:
- `crypto` and `sendMjmlEmail` are already imported at the top of the file (lines 2–3) — no new imports needed.
- Mirrors `overrideSendEmailConfirmation` (lines 533–576) for the email send block — same template, same subject, same variables, same try/catch + non-fatal error log strategy.
- Mirrors `overrideForgotPassword` (line 486) for token generation: `crypto.randomBytes(64).toString("hex")`.
- Saves the token BEFORE sending email (matches the ordering in `overrideForgotPassword` line 488–492 and the ordering comment "Save token before sending email (matches built-in ordering)").
- Returns `{ resent: true }` (200 OK) — distinct from `{ ok: true }` so the frontend can branch on it.

What NOT to do:
- Do NOT call any plugin's emailConfirmation service directly — `sendMjmlEmail` is the project-standard MJML pathway, already used by both `registerUserLocal` and `overrideSendEmailConfirmation` in this file.
- Do NOT fall back silently if the email send fails — log the error (the token is already persisted, so the user can hit `/auth/send-email-confirmation` to retry).
- Do NOT lowercase the email or re-validate it — we already have the user record by token; the email is trusted.

<task type="auto">
  <name>Task 2: Refactor /registro/activar.vue to use SSR-compatible useAsyncData with three-outcome handling</name>
  <files>apps/website/app/pages/registro/activar.vue</files>
  <action>
Rewrite the `<script setup>` block of `apps/website/app/pages/registro/activar.vue` so the API call runs during SSR via `useAsyncData`, and Swal/navigateTo only run client-side. Keep the `<template>` (the `LoadingDefault` component) UNCHANGED.

Concrete structure:

```vue
<template>
  <div class="page page--provider">
    <LoadingDefault />
  </div>
</template>

<script setup lang="ts">
import LoadingDefault from "@/components/LoadingDefault.vue";

useSeoMeta({ robots: "noindex, nofollow" });

const route = useRoute();
const apiClient = useApiClient();
const { Swal } = useSweetAlert2();

type ConfirmationResult =
  | { ok: true }
  | { resent: true }
  | { error: true };

const confirmation = route.query.confirmation as string | undefined;

const { data } = await useAsyncData<ConfirmationResult>(
  "activar-email-confirmation",
  async () => {
    if (!confirmation) {
      return { error: true };
    }
    try {
      const res = await apiClient<{ ok?: boolean; resent?: boolean }>(
        `/auth/email-confirmation?confirmation=${confirmation}`,
        { method: "GET" },
      );
      if (res?.resent) return { resent: true };
      return { ok: true };
    } catch {
      return { error: true };
    }
  },
  { default: () => ({ error: true }) },
);

onMounted(async () => {
  const result = data.value;

  if (result && "ok" in result) {
    await Swal.fire(
      "¡Cuenta confirmada!",
      "Tu correo electrónico ha sido verificado. Ya puedes iniciar sesión.",
      "success",
    );
  } else if (result && "resent" in result) {
    await Swal.fire(
      "Te hemos enviado un nuevo correo",
      "El enlace anterior ya había sido utilizado. Revisa tu correo electrónico para confirmar tu cuenta.",
      "info",
    );
  } else {
    await Swal.fire(
      "Enlace inválido",
      "El enlace es inválido o ha expirado. Solicita un nuevo correo de confirmación.",
      "error",
    );
  }

  await navigateTo("/login");
});
</script>
```

Key requirements (each enforced by the structure above):

- `useAsyncData` is the sole data-loading trigger (CLAUDE.md rule). Runs during SSR + hydrates on client.
- `default: () => ({ error: true })` provides a non-undefined fallback (CLAUDE.md rule: "always provide a default option in useAsyncData").
- Unique key `"activar-email-confirmation"` per CLAUDE.md ("useAsyncData keys must be unique per page").
- Catch errors INSIDE the useAsyncData callback and reduce them to `{ error: true }` — do NOT let `useAsyncData` throw, otherwise SSR would 500. Network failures, 400 responses, missing token all collapse to the same error outcome (per task spec: "Error thrown (invalid token, !user) → Swal error").
- Missing `route.query.confirmation` short-circuits to `{ error: true }` WITHOUT making an HTTP call (per task spec: "If it's missing, skip the API call and treat as error case").
- All side effects (Swal.fire, navigateTo) live inside `onMounted` — they are client-only by definition. No `import.meta.client` guard needed because `onMounted` does not fire on the server in Nuxt 4.
- Type narrowing via `"ok" in result` / `"resent" in result` keeps TypeScript strict — no `any`.

What to remove from the existing file:
- The `confirm()` async function declared in setup.
- The bare `confirm()` call at the bottom of `<script setup>`.

What to keep:
- The `<template>` block exactly as-is.
- The `useSeoMeta({ robots: "noindex, nofollow" })` line.
- The imports for `LoadingDefault`, `useApiClient`, `useSweetAlert2` (these are already auto-imported except `LoadingDefault`).

What NOT to do:
- Do NOT call `apiClient` outside `useAsyncData` (would cause double-fetch on SSR + hydration per CLAUDE.md anti-pattern).
- Do NOT pair `await apiClient(...)` with `useAsyncData` in the same setup (the CLAUDE.md double-fetch warning).
- Do NOT use `onMounted` for the data fetch itself — only for Swal + navigateTo.
- Do NOT throw inside the `useAsyncData` callback — reduce all failures to `{ error: true }`.
- Do NOT rename the page, change its layout, or modify the template HTML.
- Do NOT introduce `any` casts; use the discriminated union `ConfirmationResult`.
