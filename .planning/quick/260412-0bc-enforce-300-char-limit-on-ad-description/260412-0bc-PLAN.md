<objective>
Enforce a hard 300-character cap on the ad description textarea in `FormCreateTwo.vue` so that neither typing nor pasting can exceed the limit.

Purpose: Today the textarea has a Yup `.test()` that surfaces "Descripción no válida" via the counter once 300 chars are exceeded, but nothing prevents the user from typing or pasting beyond the limit. The submit button stays disabled but the input visually keeps growing, which is confusing. We previously learned (search input fix, quick task 260412-02k) that `maxlength` alone is not enough for paste in some browsers/contexts — we also need a JS-level guard.

Output: A single edited Vue component with `maxlength="300"` on the `<Field>` and a Vue `watch` on `form.description` that hard-slices to 300 chars on every change (covering paste, drag-drop, autofill, and any programmatic v-model write).
</objective>

Script (lines 180–192):
```ts
const maxChars = 300;

// Computed property for remaining characters
const remainingChars = computed(() => {
  return maxChars - form.value.description.length;
});

// Handle text area input
const handleTextArea = () => {
  if (form.value.description.length > maxChars) {
    form.value.description = form.value.description.slice(0, maxChars);
  }
};
```

Notes:
- `<Field as="textarea">` from vee-validate forwards arbitrary attrs (incl. `maxlength`) to the rendered `<textarea>`.
- `@input="handleTextArea"` already exists but only fires for native input events; programmatic v-model writes (e.g. some paste handlers, autofill, store hydration in `onMounted`) bypass it. A `watch` on `form.value.description` is the reliable enforcement point.
- The Yup schema (lines 154–159) intentionally has no `.max(300)` — the counter UX comes from `remainingChars`. Do not add `.max()` to Yup; the cap is enforced at the input layer.
