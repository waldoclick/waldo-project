<objective>
Articles are visible via the public REST API (`/api/articles`) but invisible in the Strapi admin panel (`/admin` content manager). Root cause: the Redis cache middleware (`apps/strapi/src/middlewares/cache.ts`) sets `Cache-Control: public, max-age=14400` on every GET/HEAD response — including `/content-manager/*` and `/admin/*` requests — BEFORE running the `shouldNotCache` filter. The Redis layer correctly skips caching those paths, but the browser obeys the public cache header and stores admin responses for up to 4 hours. After any article create/update/delete the admin's content manager keeps showing the stale (or empty) cached list.

Purpose: Restore admin visibility into articles (and every other admin/content-manager resource) by ensuring `Cache-Control: public` is only emitted for routes that should actually be public-cached.

Output: Patched `cache.ts` middleware that sets cache headers only after `shouldNotCache` confirms the route is cacheable. Public API caching behavior preserved.
</objective>

  <name>Task 1: Move Cache-Control header set behind shouldNotCache filter</name>
  <files>apps/strapi/src/middlewares/cache.ts</files>
  <action>
In the exported default function (the koa middleware returned at the bottom of the file), reorder the cache header logic so `Cache-Control: public` is NEVER emitted for admin/content-manager (or any path where `shouldNotCache(ctx.url)` returns true).

Current (buggy) flow inside the middleware:
1. Set `X-Cache-Middleware: active` (keep)
2. Init Redis if needed (keep)
3. If Redis unavailable AND method is GET/HEAD → set Cache-Control + X-Cache=MISS, return next() — BUG: also runs on admin paths
4. Always set Cache-Control + X-Cache=MISS for GET/HEAD — BUG: runs before shouldNotCache check
5. If shouldNotCache → return next() (too late, header already set)
6. Cache invalidation for non-GET methods
7. Redis GET attempt + body replacement on hit
8. next() then Redis SET on cache miss

Required (fixed) flow:
1. Set `X-Cache-Middleware: active`
2. Init Redis if needed
3. **NEW guard FIRST**: if `shouldNotCache(ctx.url)` is true → just `return await next()` — emit NO Cache-Control, NO X-Cache header, NO Redis interaction. This is the critical fix: admin and content-manager responses must pass through with no public cache header.
4. After shouldNotCache passes (route is cacheable):
   - For GET/HEAD: set `Cache-Control: public, max-age=${ttl}, s-maxage=${ttl}` and `X-Cache: MISS` (default; will overwrite to HIT below if cached)
   - If Redis unavailable: just `return await next()` (headers already set above for HTTP-level cache; no Redis storage)
   - For non-GET/HEAD on cacheable routes: invalidate via `invalidateCollectionCache(ctx.url)`, return next()
5. Generate cache key, attempt Redis GET. If hit → replace body, set `X-Cache: HIT` (Cache-Control already set), return.
6. On miss → next(); if response is 200 with body, store in Redis with TTL. Cache-Control already set above; no need to re-set.

Implementation notes:
- Keep `X-Cache-Middleware: active` as the very first header (useful for debugging, cheap, harmless on admin)
- Do NOT delete `shouldNotCache` rules — they are correct, just being checked too late
- Do NOT add `/admin` or `/content-manager` to `shouldNotCache` — the existing `if (!url.startsWith("/api/")) return true;` already covers them. The bug is purely about ordering.
- Preserve the existing log lines (`console.log("cachedResponse", ...)`) only if they were already there — do not add new ones
- The `getCacheTTL()` helper is unchanged — it just reads `DEFAULT_TTL`
- Per CLAUDE.md: no unused vars/imports — if `getCacheTTL` ends up referenced once, that is fine; if any helper becomes unused after the reshuffle, delete it
- Per CLAUDE.md: no `any` casts; preserve existing types (`Context`, `Promise<unknown>`)
