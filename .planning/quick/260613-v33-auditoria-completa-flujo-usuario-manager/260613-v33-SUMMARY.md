# Phase 260613-v33 Plan 01: SSR Fail-Open in auth.ts Summary

**One-liner:** SSR fail-open after fetchUser try/catch prevents Manager logout on window.location.reload() when Strapi's proxy-auth middleware rejects requests lacking X-Proxy-Key.

## What Was Built

Modified `apps/website/app/middleware/auth.ts` to add an SSR fail-open that mirrors the pattern already in `dashboard-guard.global.ts` (D-03).

**Root cause:** When `LightboxRazon.vue` calls `window.location.reload()` after deactivating an ad, the browser triggers a full SSR pass. During SSR, `useStrapiAuth().fetchUser()` calls Strapi at `API_URL` directly — without the `X-Proxy-Key` header that the Nitro proxy injects. Strapi's `proxy-auth` middleware rejects with 401. The catch block swallows the error, `user.value` remains null, and `auth.ts` redirects to `/login`.

**Fix:** After the `try/catch` block inside the `if (token.value)` branch, added:
```typescript
if (import.meta.server && !user.value) return;
```

This fail-open only fires when: (a) running on SSR, (b) a token exists, and (c) fetchUser still resulted in a null user — the exact failure scenario. Client hydration re-runs the middleware with `user.value` already populated via the Nitro proxy (which carries `X-Proxy-Key`), so the route protection holds.

**Also fixed:** Removed unused `_from` parameter from the middleware signature (CLAUDE.md / Codacy rule: never leave `_`-prefixed unused params).
