<objective>
Corregir el logout inesperado del Manager después de desactivar un aviso en /cuenta/mis-anuncios.

Purpose: El `middleware/auth.ts` no tiene SSR fail-open. Cuando `window.location.reload()` en LightboxRazon dispara una carga SSR, `useStrapiAuth().fetchUser()` llama a Strapi directo (API_URL) sin `X-Proxy-Key`. El `proxy-auth` middleware de Strapi rechaza con 401. `auth.ts` atrapa el error, `user.value` queda null, y redirige a `/login`. El patrón correcto ya existe en `dashboard-guard.global.ts` (SSR fail-open per D-03).

Output: auth.ts con SSR fail-open. Usuario manager — y todos los usuarios — no son expulsados en SSR reload.
</objective>

<task type="auto">
  <name>Task 1: Add SSR fail-open to auth.ts matching dashboard-guard pattern</name>
  <files>apps/website/app/middleware/auth.ts</files>
  <action>
    Reemplazar el middleware completo con la versión que incluye SSR fail-open.

    PROBLEMA RAÍZ: En SSR, `useStrapiAuth().fetchUser()` usa `runtimeConfig.strapi.url = API_URL`
    (Strapi directo) pero NO inyecta `X-Proxy-Key`. El middleware `proxy-auth` de Strapi rechaza
    con 401. El catch swallows el error. `user.value` queda null. `auth.ts` redirige a `/login`.

    SOLUCIÓN: Cuando el token existe pero fetchUser falla en SSR (`import.meta.server`),
    hacer fail-open (return) en lugar de redirigir. El cliente re-corre el middleware con
    `user.value` ya poblado (la hidratación usa BASE_URL → proxy Nitro → que SÍ lleva X-Proxy-Key).

    Patrón de referencia: `dashboard-guard.global.ts` línea 18:
    `if (!roleName) return; // SSR fail-open skip per D-03`

    Implementar:
    ```typescript
    export default defineNuxtRouteMiddleware(async (to) => {
      const user = useStrapiUser();
      if (!user.value) {
        const token = useStrapiToken();
        if (token.value) {
          const { fetchUser } = useStrapiAuth();
          try {
            await fetchUser();
          } catch {
            /* Strapi unavailable — treat as unauthenticated */
          }
          // SSR fail-open: fetchUser() on server uses API_URL directly without
          // X-Proxy-Key — proxy-auth middleware rejects with 401, user stays null.
          // Fail-open here; client hydration re-runs the guard with user already
          // populated via the Nitro proxy (which does carry X-Proxy-Key).
          // Pattern per D-03 (dashboard-guard.global.ts).
          if (import.meta.server && !user.value) return;
        }
      }
      if (!user.value) {
        useCookie("redirect", { path: "/" }).value = to.fullPath;
        return navigateTo("/login");
      }
    });
    ```

    NOTAS:
    - Quitar el parámetro `_from` no utilizado (Codacy flag: _-prefix unused params).
    - El `if (import.meta.server && !user.value) return;` solo aplica cuando hay token
      pero fetchUser falló → token existe, user null → SSR fail-open. Sin token sigue
      bloqueando correctamente (nunca llega al fetchUser block).
    - Si el token NO existe en SSR, el bloque interno no se ejecuta y cae al
      `if (!user.value)` → redirige al login, que es correcto (no hay sesión).
