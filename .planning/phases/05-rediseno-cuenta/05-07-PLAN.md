NEVER an aggregate counter. Owner is excluded; views are unique per visitor per day.

  <behavior>
    recordView(adDocumentId, viewerId, source, ip, ua):
    - Test 1: when the ad has no existing view for the computed visitor_hash today → creates exactly one ad-view row (ad relation set, viewed_at now, source stored, viewer = viewerId or null).
    - Test 2: when an ad-view row already exists for the same visitor_hash AND the same calendar day → does NOT create a second row (idempotent per visitor/day).
    - Test 3: when viewerId === ad.user.id (owner viewing own ad) → creates NO row and returns early.
    - Test 4: visitor_hash recipe is deterministic: sha256(`${ip}|${ua}|${yyyy-mm-dd}`) — same ip+ua on the same day yields the same hash; different day yields a different hash.
    - Test 5: a thrown error from the DB layer is swallowed (tracking must never break the ad page) — recordView resolves without rethrowing.
  </behavior>
  <implementation>
    Build in RED→GREEN order.

    1. Content type `ad-view` (schema.json, collectionName "ad_views", singularName "ad-view", pluralName "ad-views", draftAndPublish false). Attributes:
       - `ad`: relation manyToOne → api::ad.ad
       - `viewed_at`: datetime, required true
       - `visitor_hash`: string (for dedupe)
       - `source`: string (e.g. "detail", nullable)
       - `viewer`: relation oneToOne → plugin::users-permissions.user (nullable)
       Mirror the ad content-type folder layout. Add routes/ad-view.ts = `factories.createCoreRouter("api::ad-view.ad-view")`, controllers/ad-view.ts = `factories.createCoreController("api::ad-view.ad-view")`, services/ad-view.ts = `factories.createCoreService("api::ad-view.ad-view", ({ strapi }) => ({ ...custom methods }))`.

    2. Content type `ad-contact` (collectionName "ad_contacts"). Attributes:
       - `ad`: relation manyToOne → api::ad.ad
       - `type`: enumeration ["call","message"], required true
       - `visitor_hash`: string
       - `created_at`: datetime (or rely on Strapi createdAt — but per D-06 add explicit `created_at` datetime for the event timestamp)
       Same folder layout + core router/controller/service.

    3. `ad-view` service `recordView(adDocumentId, viewerId, source, ip, ua)`:
       - Compute `const day = new Date().toISOString().slice(0,10)` (yyyy-mm-dd).
       - `const visitor_hash = crypto.createHash("sha256").update(`${ip}|${ua}|${day}`).digest("hex")` (import node crypto, same pattern as ad controller's existing `import crypto from "crypto"`).
       - Resolve the ad + its owner: `strapi.db.query("api::ad.ad").findOne({ where: { documentId: adDocumentId }, populate: ["user"] })`. If not found → return.
       - OWNER EXCLUSION: `if (viewerId && ad.user?.id === viewerId) return;`
       - DEDUPE: query ad-view for an existing row with this `visitor_hash` AND `viewed_at` within today (gte start-of-day). If exists → return.
       - Else `strapi.db.query("api::ad-view.ad-view").create({ data: { ad: ad.id, viewed_at: new Date(), visitor_hash, source: source ?? "detail", viewer: viewerId ?? null } })`.
       - Wrap the whole body in try/catch; log via the existing logtail logger and swallow (never throw).

    4. Wire tracking into `apps/strapi/src/api/ad/services/ad.ts` `findBySlug(slug, userId)`: AFTER the ad is resolved, call (fire-and-forget, awaited but error-swallowed) `strapi.service("api::ad-view.ad-view").recordView(ad.documentId, userId, "detail", ip, ua)`. Obtain ip/ua: pass them down from the controller `findBySlug` (read `ctx.request.ip` and `ctx.request.headers["user-agent"]`) into the service call — extend the service signature to `findBySlug(slug, userId, meta?)` where meta = { ip, ua }, OR read them in the controller and call recordView there right after the service returns the ad. PREFER calling recordView in the CONTROLLER findBySlug (it already has ctx + userId + the returned ad.documentId) to avoid threading ip/ua through the service. Keep the existing return shape unchanged — tracking is a side effect.

    5. `ad-contact` endpoint: custom route `POST /ads/:documentId/contact` (in a new `00-ad-contact-custom.ts`, config auth optional) → controller method that reads `type` from body, computes visitor_hash like recordView, and creates an ad-contact row. (Wiring this to the actual contact buttons is Claude's discretion per D-06 — at minimum expose the endpoint + service method `recordContact(adDocumentId, type, ip, ua)`.)

    6. Test file `apps/strapi/tests/api/ad-view/ad-view.service.test.ts`: AAA, mock `strapi.db.query` with per-UID routing (return ad on api::ad.ad findOne, control ad-view findMany/create), mock crypto only if needed (prefer real crypto to assert determinism). Cover Tests 1-5 above.
