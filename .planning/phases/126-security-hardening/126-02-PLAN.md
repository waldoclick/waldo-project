<objective>
Close the HIGH mass-assignment payment/approval bypass on ad create/update. Today a client can POST/PUT to `/api/ads` and set privileged fields (`active`, `is_paid`, `banned`, `rejected`, `remaining_days`, `duration_days`, `draft`, `actived_by`, `user`) directly in the body — publishing or marking an ad paid/approved without payment or moderator approval, violating the project payment invariant. Add a `protect-ad-fields` global middleware that strips these fields from POST and PUT `/api/ads` (and `/api/ads/:id`) request bodies, mirroring the existing `protect-user-fields` middleware. These flags must only ever be set server-side by the payment flow and the manager `approveAd` path.

Purpose: Enforce that publish/paid/approval state is server-controlled, never client-settable.
Output: A new `protect-ad-fields` middleware, its registration in `config/middlewares.ts`, and a Jest regression test.
</objective>

EXACT PATTERN TO MIRROR — apps/strapi/src/middlewares/protect-user-fields.ts:
- Default export: `(_config, _context: { strapi: Core.Strapi }) => async (ctx: Context, next) => { ... await next(); }`
- A `const PROTECTED_*_FIELDS = [...] as const;` list
- A path regex test guarding the method + path
- Handles both `body.data` (Strapi wrapper) and flat body via a `stripProtectedFields(obj)` helper that deletes matching keys in place and returns the stripped names; warns via console.warn

DIFFERENCES for ads:
- Methods: POST and PUT (protect-user-fields only handled PUT)
- Path: POST /api/ads (collection) and PUT /api/ads/:id (single). The ad controller also uses { data: {...} } via core controller and AdUtils. Match both `/api/ads` (no id, for POST) and `/api/ads/:id` (numeric id, for PUT). Include an optional trailing slash so `/api/ads/` and `/api/ads/123/` are also covered (learn from the protect-user-fields trailing-slash bug being fixed in 126-04 — do not reproduce it here).

Custom ad sub-routes (e.g. /api/ads/:id/approve, /api/ads/draft, /api/ads/upload, /api/ads/count) must NOT be affected: the regex must match ONLY the bare collection (/api/ads) and the bare single-item path (/api/ads/:numericId), not deeper sub-paths. approveAd/rejectAd/bannedAd set these flags server-side via the service and must keep working.
