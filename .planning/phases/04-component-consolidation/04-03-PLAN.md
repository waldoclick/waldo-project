<!-- Analysis summary for executor reference -->

ReservationsFree.vue vs ReservationsUsed.vue differences (disqualify consolidation):
1. Shared store section key ("reservations") — both views read/write the same store key. A generic component would have both views fighting over the same pagination state, with no mechanism to distinguish "free page 3" from "used page 3".
2. Fetch strategy mismatch — ReservationsFree uses server-side pagination (Strapi handles pagination). ReservationsUsed fetches all 1,000 records and does client-side slice+filter. These cannot be unified under a single prop without an architectural refactor.
3. Column schema differs — ReservationsUsed has an extra "Anuncio" (ad.name) column (4 columns → 5 columns). A generic component would require configurable columns, approaching the complexity of keeping the files separate.
4. Business filter is part of identity — The $null/$notNull filter on the `ad` relation is not a display mode but a data split. Passing it as a prop would require arbitrary filter injection, not a simple prop.

FeaturedFree.vue vs FeaturedUsed.vue differences (disqualify consolidation):
1. Same shared store section key ("featured") — same pagination state conflict as Reservations*.
2. Business filter complexity — FeaturedFree has `{ ad: { $null: true }, price: { $eq: "0" } }`. These are not display toggles; they are domain rules.
3. Column schema differs — FeaturedUsed has an extra "Anuncio" column (4 columns → 5 columns).
4. Both have onMounted double-fetch bug (outside Phase 4 scope).

Contrast with AdsTable pattern (which did qualify for consolidation):
- All 6 Ads* components shared a single endpoint pattern with one variable (the endpoint string)
- All 6 shared identical column schemas
- All 6 had independent store section keys (adsPendings, adsActives, etc.)
- The only behavioral difference was one conditional button (showWebLink)

The Reservations*/Featured* pairs fail all three of these conditions.

Recommended future action: If these are to be consolidated, the prerequisite is:
- Add separate store section keys for the "free" and "used" sub-views
- Align fetch strategies (move ReservationsUsed to server-side pagination)
- Then the column differences can be handled via a columns prop as a subsequent step
This would be appropriate for a future technical debt phase.
