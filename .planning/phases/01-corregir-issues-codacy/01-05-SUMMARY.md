# Phase 01 Plan 05: Defensive Hardening (open-redirect + path-traversal) Summary

Applied the 2 VALIDATE hardenings from RESEARCH Group 4: an OAuth provider allowlist guard in `useProviders` (flipping the Wave 0 RED test GREEN) and an `os.tmpdir()` confinement assertion on every `image-uploader` fs read/write. Neither was exploitable today (all call sites pass literals; filepath is server-generated by the multipart parser), but both now carry a citable in-code guard so the corresponding Codacy findings can be suppressed in plan 06, and the vector is closed against future call sites.

## What Was Built

- **Task 1 (366b4e45):** Added module-level `const ALLOWED_PROVIDERS: readonly string[] = ["google", "facebook"]` and guarded `redirectToProvider` with `if (!ALLOWED_PROVIDERS.includes(provider)) return;` before assigning `window.location.href`. `loginWithPopup` untouched. No `any`, no unused vars. Wave 0 disallowed-provider test (`redirectToProvider('evil://x')` must not navigate) now GREEN; the `google` navigation test still passes.
- **Task 2 (0e16cbcf):** Added `import os from "os"` and a `resolveUploadPath(filepath)` helper that asserts `fs.realpathSync(filepath)` starts with `fs.realpathSync(os.tmpdir()) + path.sep`, throwing `"Upload filepath outside temp dir"` otherwise and returning the resolved path. Applied to all three fs functions — `convertToWebP`, `processGalleryImage`, `processAvatarImage` — replacing `f.filepath` with the confined `resolved` path at every `fs.readFileSync` / `fs.writeFileSync` site. No `any`, no unused vars.

## Deviations from Plan

None — plan executed as written. Two intent-preserving adjustments aligned with this plan's own `<important>`/`<success_criteria>` over the action-block pseudocode:

1. **useProviders allowlist typing:** typed `ALLOWED_PROVIDERS` as `readonly string[]` and called `includes(provider)` with no cast (instead of the action block's `includes(provider as (typeof ALLOWED_PROVIDERS)[number])`). ## Residual Risk (noted, non-blocking)

The confinement `throw` lands inside the existing `catch`, so a path outside tmpdir degrades to "image silently unprocessed," not a 500 — acceptable confinement. This rests on RESEARCH's authority that Strapi's multipart parser writes uploads under `os.tmpdir()`. If Strapi were configured with a custom upload tmp dir, legitimate uploads would silently skip processing; RESEARCH asserts tmpdir, so proceeding is correct.
