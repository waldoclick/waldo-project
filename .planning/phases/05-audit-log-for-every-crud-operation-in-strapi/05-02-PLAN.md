3. Admin-panel write → admin::user row:
   - In Content Manager, create a new entry in any editable content-type (e.g. add a FAQ or a Term), then edit it, then delete it.
   - Refresh Audit Log. Expected: three new rows — action create, update, delete — for that content-type's UID, each with `actor_type = admin::user` and `actor_id` = your admin user id. `record_id` / `record_document_id` populated.

   - Expected: an audit row with `actor_type = plugin::users-permissions.user` and `actor_id` = that user's id.

5. No recursion / no runaway growth:
   - Confirm that creating audit rows did NOT itself spawn additional audit rows (row count increases by the number of real writes, not exponentially). E.g. the 3 admin writes in step 3 add exactly 3 rows, not more.

6. Business writes still succeed:
   - Confirm the create/update/delete in step 3 all completed normally (no 500s, entries saved/removed as expected) — proving the audit hook does not break real writes even if something in it fails.
