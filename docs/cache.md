# Cache System (Strapi)

Strapi implements an optional Redis-based HTTP cache via the `global::cache` middleware. It is disabled by default and activates only when `REDIS_ENABLED=true` is set in the environment.

---

## Architecture

The cache sits as a Koa middleware in the Strapi request pipeline, registered **after** all built-in Strapi middlewares and just before `global::protect-user-fields`. It intercepts HTTP requests and responses at the transport layer — it does not use any Strapi service layer or content-manager APIs.

```
Request
   │
   ▼
[strapi built-ins]
   │
   ▼
[global::cache]  ◄── Redis read/write happens here
   │
   ▼
[global::protect-user-fields]
   │
   ▼
[Strapi controller / route handler]
   │
   ▼
Response
```

Source: `apps/strapi/src/middlewares/cache.ts`  
Registration: `apps/strapi/config/middlewares.ts` (line 71)

---

## Redis Connection

The middleware uses `ioredis` and initialises lazily on first request. If Redis is unreachable, the middleware fails gracefully: requests fall through to the normal Strapi handler and HTTP cache headers are still set on GET/HEAD responses.

Connection parameters:

| Variable | Default | Purpose |
| --- | --- | --- |
| `REDIS_HOST` | `localhost` | Redis server hostname |
| `REDIS_PORT` | `6379` | Redis server port |
| `REDIS_PASSWORD` | _(empty)_ | Redis password |
| `REDIS_TTL` | `14400` (4 h) | Cache TTL in seconds |

Retry strategy: up to 3 attempts with exponential backoff (100 ms, 200 ms, 300 ms). After 3 failures the connection is abandoned and the `redis` reference is set to `null`, triggering a fresh `initRedis()` attempt on the next request.

## What Gets Cached

Only `GET` and `HEAD` requests are eligible for Redis caching. The following paths are **excluded** and always bypassed:

| Excluded pattern | Reason |
| --- | --- |
| `/admin`, `/api/admin`, `/content-manager` | Strapi admin panel |
| `/api/connect`, `/api/auth`, `**/callback` | Authentication routes |
| `/api/orders` | Order data (transactional) |
| `/api/users` | User-specific data |
| `**/uploads` | Static file uploads |
| Files with extensions `.jpg`, `.png`, `.pdf`, etc. | Binary assets |

Everything else (public API routes such as `/api/ads`, `/api/categories`, etc.) is eligible for caching.

## Cache Key Format

```
cache:<METHOD>:<url+path>:<JSON-encoded query params>
```

Example:

```
cache:GET:/api/ads?filters[status]=active:{"filters":{"status":"active"}}
```

The key is generated by `generateCacheKey(ctx)` and always includes the full URL and serialised query string, so two requests that differ only in query parameters produce different cache keys.

---

## Cache Invalidation

On `POST`, `PUT`, `PATCH`, or `DELETE` requests, the middleware calls `invalidateCollectionCache(url)` **before** passing the request to the controller. It:

1. Extracts the first two path segments (e.g. `/api/ads` from `/api/ads/abc123`).
2. Runs a Redis `KEYS cache:*:/api/ads*` scan.
3. Deletes all matching keys in a single `DEL` call.

This means any write to a collection flushes **all cached GET responses for that collection**, including paginated and filtered variants.

---

## HTTP Response Headers

The middleware sets headers on every `GET`/`HEAD` response, whether or not Redis is available:

| Header | Value |
| --- | --- |
| `X-Cache-Middleware` | `active` — confirms the middleware ran |
| `Cache-Control` | `public, max-age=<TTL>, s-maxage=<TTL>` |
| `X-Cache` | `HIT` if served from Redis, `MISS` otherwise |

> **Known limitation:** the Nuxt proxy layer does not forward `X-Cache` or `Cache-Control` headers to the browser. These headers are visible from direct Strapi requests but not from website/dashboard clients. See project memory `project_proxy_cache_headers.md`.

## Enabling in Production

1. Set `REDIS_ENABLED=true` in `apps/strapi/.env`.
2. Set `REDIS_HOST`, `REDIS_PORT`, and `REDIS_PASSWORD` to point at your Redis instance.
3. Optionally tune `REDIS_TTL` (default 4 hours).

No code changes are required — the middleware is already registered and gated on the env var.
